# Q37. Wi-Fi Security for IoT

Wi-Fi-enabled IoT devices are vulnerable to several security threats if proper protection is not implemented. One common attack vector is the **KRACK (Key Reinstallation Attack)**, which exploits weaknesses in the WPA2 four-way handshake to manipulate encryption keys and potentially allow attackers to intercept or modify network traffic. Another common attack is the **Evil Twin attack**, where an attacker creates a fake Wi-Fi access point with the same name as a legitimate network. IoT devices may unknowingly connect to the rogue network, allowing attackers to capture sensitive information. Devices are also vulnerable to **deauthentication attacks**, where attackers repeatedly disconnect devices from the network, affecting communication and availability.

To improve the security of **ESP32** Wi-Fi connections, developers should use **WPA3** whenever it is supported, as it offers stronger authentication and better protection against password-based attacks than WPA2. Using **TLS/SSL encryption** with **certificate pinning** helps ensure that the ESP32 communicates only with trusted servers, reducing the risk of man-in-the-middle attacks. Additional security measures include keeping firmware updated, using strong and unique Wi-Fi passwords, and isolating IoT devices on a separate network or VLAN.

Hardcoding Wi-Fi passwords or API keys directly into IoT source code is dangerous because anyone who gains access to the source code or firmware can extract these credentials. If the code is uploaded to a public repository, attackers may gain unauthorized access to the network or cloud services. A safer approach is to store credentials in a separate configuration file or use secure credential storage mechanisms.
